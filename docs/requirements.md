# 量化交易系统 MVP 需求文档

## 1. 项目目标

构建一个面向加密货币市场的量化交易系统，第一阶段以回测与模拟盘为核心，完成从数据采集、策略执行、风控、订单管理、前端可视化到 Docker 化部署的完整闭环，并为后续接入实盘交易预留扩展能力。

本项目的目标不是直接做高频或多交易所复杂平台，而是先落地一个可部署、可演示、可持续演进的量化交易 MVP。

## 2. 已确认的默认决策

- 交易市场：加密货币
- 实施顺序：回测 -> 模拟盘 -> 后续扩展实盘
- 部署形态：单机部署，基于 Docker Compose
- 策略范围：单策略框架，支持 1-2 个示例策略
- 接入范围：单交易所优先，建议首期默认接入 Binance
- 用户角色：单管理员，不做复杂多租户和 RBAC

## 3. 项目范围

### 3.1 本期必须包含

1. 交易核心后端
2. 策略执行与管理能力
3. 回测能力
4. 模拟盘能力
5. 风控能力
6. 数据存储与查询能力
7. 前端可视化控制台
8. REST API 与必要的实时推送能力
9. Docker 打包与 Docker Compose 编排
10. 面向云服务器部署的文档

### 3.2 本期不包含

- 高频交易
- 多交易所同时接入
- 多用户权限系统
- 复杂策略市场组合优化平台
- Kubernetes 集群部署
- AI 自动生成策略
- 直接将实盘交易作为首期验收前提

说明：虽然首期不以实盘为验收前提，但系统架构必须预留实盘接入能力，包括交易所适配层、账户同步、下单接口、密钥配置和风控拦截能力。

## 4. 目标用户与使用场景

### 4.1 目标用户

- 单个量化交易操作者
- 需要在云服务器上长期运行系统的个人开发者或研究者

### 4.2 核心使用场景

1. 配置交易对、时间周期、策略参数
2. 拉取历史行情并执行回测
3. 查看回测结果、收益曲线和关键指标
4. 启动模拟盘策略并观察订单、持仓、资产变化
5. 在前端查看系统日志、风控状态和策略运行状态
6. 在云服务器上通过 Docker Compose 一键启动系统

## 5. 功能需求

### 5.1 后端交易核心

后端需要提供完整的交易基础设施，而不是只提供一个简单脚本。

必须具备以下能力：

- 行情数据接入与标准化
- 交易所适配层封装，避免业务逻辑直接依赖某一交易所 SDK
- 账户信息同步，包括余额、可用资金、持仓等
- 订单生命周期管理，包括创建、提交、撤单、状态更新、成交记录
- 持仓与资金快照记录
- 策略执行调度与运行状态管理
- 风控拦截与异常停止机制
- 系统启动后的状态恢复能力

建议后端按模块划分为：

- API 服务
- 策略执行 Worker
- 数据采集模块
- 交易执行模块
- 风控模块
- 回测模块

### 5.2 策略系统

策略系统需要支持最小可用但可扩展的策略框架。

必须包含：

- 策略注册与管理
- 单策略运行模型
- 策略参数配置
- 交易对与时间周期配置
- 启动、停止、重启策略
- 策略运行日志记录
- 支持至少 1-2 个示例策略

示例策略建议：

- 双均线策略
- 突破或均值回归类策略二选一

要求：

- 策略逻辑与交易所适配逻辑解耦
- 策略可以复用同一套订单、风控、数据接口
- 后续能够平滑扩展到实盘执行

### 5.3 回测系统

回测是本项目首期的核心能力之一。

必须支持：

- 选择交易对、时间周期、开始时间、结束时间
- 配置策略参数
- 设置初始资金
- 设置手续费与滑点
- 执行回测并保存结果
- 返回交易明细与收益曲线

至少输出以下指标：

- 总收益率
- 年化收益率或区间收益说明
- 最大回撤
- 胜率
- 交易次数
- 盈亏比或 Profit Factor
- 资金曲线

### 5.4 模拟盘系统

模拟盘系统需要在不承担真实资金风险的前提下，验证策略执行闭环。

必须支持：

- 使用实时或准实时行情驱动策略
- 模拟账户余额
- 模拟下单撮合或状态变更
- 记录模拟订单、成交、持仓、收益
- 在前端实时查看运行状态
- 支持手动停止策略

要求：

- 模拟盘的数据结构尽量与未来实盘保持一致
- 下单和持仓逻辑与回测区分，但风控规则尽量复用

### 5.5 风控系统

风控必须作为独立能力存在，不能散落在策略代码中。

首期至少支持：

- 单笔下单金额或仓位上限
- 单策略最大持仓限制
- 最大连续亏损限制
- 止损与止盈阈值
- 单日最大亏损限制
- 异常熔断，例如数据异常、接口异常、重复下单时自动暂停策略

要求：

- 每次下单前执行风控校验
- 风控命中后记录日志并在前端显示
- 风控参数可配置

### 5.6 数据层

系统需要具备可查询、可追踪、可回放的数据基础。

首期至少存储以下数据：

- K 线历史数据
- 策略配置与运行记录
- 订单数据
- 成交数据
- 持仓快照
- 资金快照
- 回测结果
- 系统日志与告警日志

要求：

- 支持按交易对、时间区间查询
- 支持前端图表和列表展示
- 为后续分析和排障保留数据

### 5.7 前端界面

前端不能只是简单管理页，需要具备较强的信息可视化能力，并且视觉上具有专业交易控制台风格。

首期必须包含以下页面或模块：

1. 总览仪表盘
   - 总资产
   - 模拟收益
   - 策略运行状态
   - 最近订单与告警

2. 策略管理页
   - 策略列表
   - 参数编辑
   - 启动/停止操作
   - 当前状态展示

3. 回测页
   - 回测参数配置
   - 回测执行入口
   - 收益曲线
   - 指标面板
   - 交易记录列表

4. 模拟盘页
   - 当前资产
   - 当前持仓
   - 当前订单
   - 实时收益变化
   - 策略运行日志

5. 订单与成交页
   - 历史订单
   - 成交明细
   - 筛选与检索

6. 日志与告警页
   - 系统日志
   - 策略日志
   - 风控日志
   - 错误告警

前端要求：

- 支持桌面端优先，同时兼容移动端查看
- 界面风格统一，避免通用后台模板感过强
- 图表展示清晰，适合金融数据查看
- 支持实时数据刷新或 WebSocket 推送

### 5.8 API 与实时通信

后端需要提供前端可调用接口，并在必要处提供实时更新。

至少包含：

- 策略管理 API
- 回测任务 API
- 订单与成交查询 API
- 持仓与资产查询 API
- 日志查询 API
- 系统健康检查 API
- 实时推送通道，用于策略状态、订单变化、日志或资产变化展示

### 5.9 认证与安全

虽然首期是单管理员系统，但仍需具备基础安全能力。

至少包含：

- 管理后台登录能力
- API 访问鉴权
- 敏感配置通过环境变量注入
- 不在仓库中明文提交 API Key 或密钥
- 关键操作保留日志

如果后续接入实盘，需要补充：

- API Key 加密存储或安全注入方案
- 更严格的操作审计
- 可选的二次确认机制

## 6. 非功能需求

### 6.1 可维护性

- 后端模块边界清晰
- 策略、风控、交易执行解耦
- 配置与代码分离
- 日志清晰，便于排查问题

### 6.2 可扩展性

- 可以从单交易所扩展到多交易所
- 可以从模拟盘扩展到实盘
- 可以从单策略扩展到多策略
- 可以新增指标、风控规则和图表模块

### 6.3 稳定性

- 服务重启后能够恢复必要状态
- 核心异常不会导致整个系统静默失效
- 关键任务和失败信息可追踪

### 6.4 性能

- 首期不追求高频低延迟
- 适配分钟级或秒级策略场景
- 前端图表与查询在常规数据量下保持可用和流畅

### 6.5 可观测性

- 提供结构化日志或至少可清晰检索的日志
- 提供服务健康检查
- 支持查看关键运行状态

## 7. 推荐技术方案

以下为推荐实现方案，可作为后续技术设计默认基线：

### 7.1 后端

- Python 3.12
- FastAPI 作为 API 服务框架
- SQLAlchemy + Pydantic 作为数据建模与接口层基础
- PostgreSQL 作为主数据库
- Redis 作为缓存、任务状态或消息辅助组件
- 交易所接入优先考虑 ccxt 统一封装

### 7.2 前端

- Next.js 或 React 作为前端框架
- Tailwind CSS 或等效方案负责界面样式
- ECharts 或 Lightweight Charts 用于金融图表展示
- 通过 REST API + WebSocket 与后端交互

### 7.3 部署

- Docker + Docker Compose
- Nginx 或 Caddy 作为反向代理
- 云服务器默认目标环境为 Ubuntu 22.04/24.04

## 8. 交付物要求

项目完成后至少应交付：

1. 后端服务代码
2. 前端服务代码
3. Dockerfile 与 docker-compose.yml
4. 环境变量模板文件，例如 `.env.example`
5. 需求文档
6. 部署文档
7. 运行说明文档
8. 基础测试或最小可验证测试方案

## 9. 部署文档要求

后续需要单独生成部署文档，至少覆盖以下内容：

- 云服务器准备要求
- Docker 与 Docker Compose 安装
- 项目代码获取方式
- 环境变量配置说明
- 数据库初始化方式
- 一键启动命令
- 服务检查方式
- 常见故障排查
- 日志查看方式
- 升级发布方式
- 数据备份建议

## 10. 验收标准

当满足以下条件时，可认为首期 MVP 达到验收标准：

1. 系统可以通过 Docker Compose 在单台服务器启动
2. 前端可正常访问，并能展示资产、策略、订单、日志等核心信息
3. 用户可以在前端创建或配置策略参数
4. 用户可以发起回测并查看回测结果、收益曲线和交易记录
5. 用户可以启动模拟盘策略并观察订单、持仓、资金变化
6. 风控规则可以在下单前生效，并记录拦截结果
7. 服务重启后，系统能恢复基础配置和历史数据
8. 部署文档可支持在新的云服务器上完成部署

## 11. 后续扩展方向

首期完成后，可按以下顺序继续扩展：

1. 接入实盘交易
2. 增加第二家交易所
3. 增加多策略并行运行能力
4. 增加更丰富的绩效分析指标
5. 增加告警通知能力，例如 Telegram 或邮件
6. 增加更完善的权限与审计系统

## 12. 当前结论

本项目第一阶段的定位是：

构建一个面向加密货币的量化交易 MVP，优先完成回测与模拟盘闭环，具备交易核心后端、可视化前端、Docker 化部署能力，以及可落地的云服务器部署文档，并为后续实盘接入预留架构基础。
